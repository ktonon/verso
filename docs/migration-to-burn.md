# Migration to Burn: Pure Rust ML Training

## Decision

Migrate from Python/PyTorch to pure Rust with [Burn](https://burn.dev) (v0.20). See [rust-vs-python-ml.md](rust-vs-python-ml.md) for the tradeoffs analysis.

## Current State

```
erd_symbolic/     Rust   Rule engine, tokenization, validation, data generation
erd_training/     Rust   Burn model, supervised training, REINFORCE training
```

## Target State

```
erd_symbolic/     Rust   Rule engine, tokenization, validation, data generation (unchanged)
erd_training/     Rust   Burn model, supervised training, REINFORCE training (new)
```

The Python `erd_training/` directory is deleted entirely. A new Rust crate `erd_training` joins the workspace.

## Architecture

### What Burn provides

| Need | Burn API |
|------|----------|
| Transformer encoder | `burn::nn::transformer::TransformerEncoder` |
| Transformer decoder | `burn::nn::transformer::TransformerDecoder` |
| Embeddings | `burn::nn::Embedding` |
| Linear projection | `burn::nn::Linear` |
| Cross-entropy loss | `burn::nn::loss::CrossEntropyLoss` |
| AdamW optimizer | `burn::optim::AdamW` |
| Gradient clipping | `GradientClippingConfig::Norm(1.0)` |
| Autograd | `Autodiff<B>` backend decorator |
| GPU (Metal) | `burn-wgpu` backend with CubeCL |
| Checkpointing | `Recorder` trait, file-based persistence |
| Data loading | `burn::data::dataloader::DataLoaderBuilder` |

### What we write

| Component | Source |
|-----------|--------|
| `SimplificationModel` | Translate from `model.py` |
| `SupervisedTrainer` | Translate from `train.py` |
| `RLTrainer` | Translate from `rl_train.py` |
| `EncoderVocab` / `DecoderVocab` | Translate from `vocab.py` |
| `SimplificationDataset` / `SimplificationBatcher` | Translate from `dataset.py` |
| Evaluation metrics | Translate from `evaluate.py`, but call `validate_action_sequence` directly |
| Cosine schedule with warmup | ~20 lines |
| Autoregressive generate/sample | Translate from `model.py` methods |

### What disappears

| Eliminated | Why |
|-----------|-----|
| `subprocess.run()` to validate binary | Direct function call to `validate_action_sequence()` |
| `vocab.json` export/import | `IndexedRuleSet` and token lists used directly as Rust types |
| JSONL serialization for validation | Rust structs passed directly |
| Python virtualenv, pip, PYTHONPATH | Single `cargo build` |
| ONNX export (planned M7) | Inference is `model.forward()` in the same binary |

## Migration Phases

### Phase 1: Scaffold + Vocabulary + Data Loading

**Goal**: Load training data into Burn tensors. Verify token counts and example counts match the Python pipeline.

**New files**: `Cargo.toml`, `lib.rs`, `config.rs`, `vocab.rs`, `dataset.rs`

#### 1a. Workspace setup

Add `erd_training` to the workspace `Cargo.toml`. Create the crate with Burn dependencies.

#### 1b. Vocabulary

`EncoderVocab` and `DecoderVocab` replace `vocab.py`. Instead of reading `vocab.json`, they construct vocabularies directly from `erd_symbolic` types:

```
EncoderVocab:
  PAD = 0
  token_0 = 1, token_1 = 2, ...
  Built from: training_data::ALL_ENCODER_TOKENS (already exists as token_to_string)

DecoderVocab:
  PAD = 0, BOS = 1, STOP = 2
  RULE_0..RULE_n = 3..3+total_directions
  POS_0..POS_m = 3+total_directions..end
  Built from: IndexedRuleSet::total_directions + max_positions
```

No `vocab.json` needed. The vocabularies are derived from the same Rust types that generate training data, eliminating the sync problem entirely.

#### 1c. Dataset + Batcher

`SimplificationDataset` implements Burn's `Dataset` trait. It loads JSONL files (same format as today, generated by `gen_data`), parses examples, and encodes tokens to integer IDs using the vocabularies.

`SimplificationBatcher` implements Burn's `Batcher` trait. It pads sequences, creates masks, and produces batch tensors:

```
Input:  Vec<SimplificationItem>
Output: SimplificationBatch {
    enc_ids:     Tensor<B, 2, Int>,    // [B, S_enc]
    enc_pad_mask: Tensor<B, 2, Bool>,  // [B, S_enc]
    dec_input:   Tensor<B, 2, Int>,    // [B, S_dec]
    dec_target:  Tensor<B, 2, Int>,    // [B, S_dec]
    dec_pad_mask: Tensor<B, 2, Bool>,  // [B, S_dec]
}
```

#### Verification

- `EncoderVocab.size()` matches Python's 76 (75 tokens + PAD)
- `DecoderVocab.size()` matches Python's ~320 (PAD + BOS + STOP + ~253 rules + 64 positions)
- Dataset loads all ~14.7K examples from 3 JSONL files
- Train/val split with seed=42 produces same split sizes
- Batch tensor shapes match Python's `collate_fn` output

### Phase 2: Model

**Goal**: Build the transformer encoder-decoder in Burn. Verify parameter count matches ~1.4M.

**New files**: `model.rs`

#### Architecture

```
SimplificationModel<B: Backend> {
    // Encoder
    enc_tok_emb:  Embedding<B>               // enc_vocab_size x d_model
    enc_pos_emb:  Embedding<B>               // max_enc_len x d_model
    encoder:      TransformerEncoder<B>      // 4 layers, 4 heads, d_ff=256

    // Decoder
    dec_tok_emb:  Embedding<B>               // dec_vocab_size x d_model
    dec_pos_emb:  Embedding<B>               // max_dec_len x d_model
    decoder:      TransformerDecoder<B>      // 4 layers, 4 heads, d_ff=256

    // Output
    output_proj:  Linear<B>                  // d_model -> dec_vocab_size
}
```

Burn's `TransformerEncoderConfig` and `TransformerDecoderConfig` accept `d_model`, `d_ff`, `n_heads`, `n_layers`, `dropout`.

#### Forward pass

```
fn forward(enc_ids, dec_input, enc_pad_mask, dec_pad_mask) -> Tensor<B, 3>:
    enc_emb = enc_tok_emb(enc_ids) + enc_pos_emb(positions)
    memory  = encoder.forward(TransformerEncoderInput::new(enc_emb).mask_pad(enc_pad_mask))

    dec_emb = dec_tok_emb(dec_input) + dec_pos_emb(positions)
    output  = decoder.forward(TransformerDecoderInput::new(dec_emb, memory.context)
                .mask_pad(dec_pad_mask)
                .mask_memory(enc_pad_mask))

    output_proj(output)
```

Burn's `TransformerDecoder` generates the causal mask internally (autoregressive by default).

#### generate() and sample()

Autoregressive decoding methods, translated directly from `model.py`. These run in `no_grad` context (use the inner backend, not `Autodiff`).

- `generate()`: argmax at each step
- `sample()`: multinomial sampling with temperature

If Burn lacks `multinomial`, implement via: compute cumulative softmax, sample uniform random, binary search for the index. This is ~15 lines.

#### Verification

- Parameter count within 5% of 1,430,549 (small differences possible due to Burn's layer norm implementation)
- Forward pass on a dummy batch produces logits of shape `[B, S_dec, dec_vocab_size]`
- `generate()` on random input produces token sequences that terminate at STOP
- `sample()` with temperature=0.01 approximates `generate()` (nearly deterministic)

### Phase 3: Supervised Training

**Goal**: Train Phase 1 (supervised) and match Python's training loss trajectory.

**New files**: `train.rs`, `schedule.rs`, `bin/train.rs`

#### Training loop

Custom loop (not Burn's `Learner`). The Learner abstraction doesn't fit our needs because:
1. We need the same loop structure for RL training (Phase 5)
2. We need custom validation that calls `validate_action_sequence`
3. Our logging format should match the Python output for comparison

```
for epoch in 0..max_epochs:
    for batch in train_dataloader:
        logits = model.forward(batch)
        loss = cross_entropy(logits.reshape(-1, V), targets.reshape(-1))
        grads = loss.backward()
        grads = clip_grad_norm(grads, 1.0)
        model = optimizer.step(lr, model, grads)
        scheduler.step()

    val_loss = validate(model, val_dataloader)
    if val_loss < best: save_checkpoint(model)
    if patience_exhausted: break
```

#### Cosine schedule with warmup

```
fn lr_at_step(step, warmup_steps, total_steps, base_lr) -> f64:
    if step < warmup_steps:
        base_lr * step / warmup_steps
    else:
        let progress = (step - warmup_steps) / (total_steps - warmup_steps)
        base_lr * 0.5 * (1.0 + cos(pi * progress))
```

#### Checkpointing

Use Burn's `NamedMpkFileRecorder` (named MessagePack format) for compact, fast checkpoints. Save model state + optimizer state + epoch + config.

#### Configuration

All hyperparameters exposed as CLI args via `clap`:

```bash
cargo run --release --bin train -- \
    --data-dir data_training \
    --batch-size 64 \
    --lr 3e-4 \
    --max-epochs 100 \
    --device wgpu  # or cpu
```

#### Verification

- Training loss at epoch 1 is in the same ballpark as Python (~3-4)
- Loss decreases monotonically for the first 10 epochs
- Final train loss ~0.20, val loss ~0.53 (matching Python's M4 results)
- Training completes in comparable time (~39s/epoch on CPU, faster if validation overhead eliminated)
- Checkpoint saves and loads correctly (train for 5 epochs, load, continue training)

### Phase 4: Evaluation

**Goal**: Evaluate trained model by calling `validate_action_sequence` directly. No subprocess.

**New files**: `evaluate.rs`, `bin/evaluate.rs`

#### Direct validation

This is the core architectural win. In Python:

```python
# Write predictions to temp file
# subprocess.run(["target/release/validate", ...])
# Parse stdout JSON
```

In Rust:

```rust
let result = validate_action_sequence(&expr, &actions, &indexed_rules);
```

One function call. No serialization. No process spawn. No I/O.

#### Metrics

```rust
struct EvalMetrics {
    total_examples: usize,
    fully_valid_count: usize,
    validity_rate: f64,
    mean_valid_fraction: f64,
    mean_complexity_delta: f64,
    mean_reward: f64,
    improved_count: usize,
    unchanged_count: usize,
    worsened_count: usize,
    empty_sequence_count: usize,
    mean_steps: f64,
}
```

#### Verification

- Evaluate the Burn-trained checkpoint on the same 1469 validation examples
- Validity rate, complexity delta, improved/worsened counts should match Python's M5 results (within noise from random init differences)
- Evaluation completes faster than Python (no subprocess overhead)

### Phase 5: REINFORCE Training

**Goal**: Translate `rl_train.py` to Rust. The subprocess bottleneck that motivated this entire migration is eliminated.

**New files**: `rl_train.rs`, `bin/rl_train.rs`

#### Training step

```
fn rl_train_step(model, batch, rules, config, baseline):
    // 1. Encode input expressions
    enc_ids, enc_pad_mask = encode_batch(batch)

    // 2. Sample action sequences (no grad — use inner backend)
    sampled_ids = model.sample(enc_ids, temperature)

    // 3. Decode sampled tokens to actions
    actions = decode_action_sequences(sampled_ids, dec_vocab)

    // 4. Validate — DIRECT FUNCTION CALL (the whole point)
    results = batch.iter()
        .map(|ex| validate_action_sequence(&ex.expr, &actions[i], &rules))
        .collect()

    // 5. Compute rewards
    rewards = results.iter().map(compute_reward).collect()

    // 6. Forward pass WITH grad
    logits = model.forward(enc_ids, dec_input, masks)
    log_probs = log_softmax(logits).gather(targets)

    // 7. REINFORCE loss
    advantages = normalize(rewards - baseline)
    loss = -(log_probs / seq_lengths * advantages).mean()
            - entropy_bonus * mean_entropy(logits)

    // 8. Backward + update
    grads = loss.backward()
    model = optimizer.step(lr, model, grads)

    // 9. Update baseline (EMA)
    baseline = decay * baseline + (1 - decay) * mean_reward
```

#### Performance impact

The validation step (step 4) was the bottleneck in Python — a subprocess call per batch. Now it's a direct function call that processes 32 examples in microseconds. Expected speedup: **5-10x per batch** (dominated by model forward/backward, not validation).

This also means **GPU training should actually help** now. The I/O bottleneck that made MPS 2x slower than CPU is gone. The model's forward/backward passes are pure compute — exactly what GPUs excel at.

#### Verification

- Mean reward increases across epochs (matching Python's +0.889 -> +1.335 trajectory)
- Validity rate is comparable (~49-53% for sampling)
- No entropy collapse (entropy bonus = 0.05 prevents it)
- Per-epoch time is significantly faster than Python's 483-694s
- GPU (Metal via wgpu) is now faster than CPU (the I/O bottleneck is gone)

## Risk Mitigation

### Burn API gaps

**Risk**: Burn's transformer API may not support padding masks or cross-attention masks exactly like PyTorch.

**Mitigation**: Burn's `TransformerEncoderInput` and `TransformerDecoderInput` have `.mask_pad()` and `.mask_memory()` methods. If the built-in transformer modules don't fit, we can build from `MultiHeadAttention` + `Linear` + `LayerNorm` directly — more code but full control.

### Multinomial sampling

**Risk**: Burn may not have `torch.multinomial` equivalent.

**Mitigation**: Implement via cumulative sum + uniform random + searchsorted. This is ~15 lines of tensor ops and is well-understood.

### Metal backend stability

**Risk**: CubeCL's Metal backend is new (added in 0.20, Jan 2026).

**Mitigation**: Start development with `ndarray` (CPU) backend. Switch to `wgpu` (Metal) once training is verified correct on CPU. The model is generic over `Backend`, so this is a one-line change.

### Compile times

**Risk**: Burn with GPU backends has long compile times.

**Mitigation**: All hyperparameters are CLI args (no recompilation for tuning). Use `cargo build --release` once, then run the binary. Incremental compilation for code changes is fast (~10-20s).

### Training reproducibility

**Risk**: Burn and PyTorch may not produce identical results for the same architecture.

**Mitigation**: We don't need identical results. We need the same quality: train loss ~0.20, val loss ~0.53, validity >65%. Small differences from different floating-point ordering, initialization, and RNG are expected and acceptable.

## Verification Strategy

Each phase has a clear verification step. The overall migration is verified by:

1. **Supervised training** matches Python's M4 results (train loss, val loss)
2. **Evaluation** matches Python's M5 results (validity rate, complexity delta)
3. **RL training** matches Python's M6 results (reward trajectory, validity rate)
4. **Performance**: RL training is significantly faster (no subprocess bottleneck)
5. **GPU utilization**: Metal backend is now faster than CPU (I/O bottleneck eliminated)

If any phase fails verification, we can debug against the Python implementation which remains in git history.
